## 4. Single-writer assumptions

- **No DB has writers in more than one OS process.** PZService (single
  uvicorn, `--loop asyncio`) is the only registered service; no scheduled
  tasks launch workers; two operator-run seed/apply tool scripts would
  contend with the live service only if run manually while it's up.
- **Multi-CONTEXT (handlers + APScheduler thread) writers exist** on:
  customer_master.sqlite (5 writer modules), proforma_links.db (enrichment
  processor on the scheduler + many handlers), contractor_poll.db (Run Now
  endpoint + 30s tick). These three combine multi-context writes with NO
  WAL + NO timeout — the exact "operator hits Run Now while the tick fires"
  lock-error scenario.
- **EJ_ orphan reservation worker: DORMANT confirmed** — worker_tick()
  defined (reservation_worker.py:481), zero call sites, no subprocess/task/
  service launches it; only explicit HTTP endpoints touch rworker functions.

## 6. Crash-recovery posture

- WAL DBs (most): power loss loses at most the last un-checkpointed commits'
  durability window under synchronous=NORMAL — consistent recovery, no
  corruption expected.
- DELETE-journal DBs (proforma_links, payment_state, contractor_poll):
  synchronous=FULL default → durable but writer-blocking; consistent
  rollback recovery.
- Service: NSSM auto-restart on any exit (AppExit=Restart, 3s delay, 10s
  throttle), Start=automatic, stdout/stderr redirected with online rotation
  at 10MB. Restart posture is good.
- Live log tail shows the service healthy (30s scheduler ticks) — and one
  REAL latent prod bug: `routes_dhl_clearance [lane-a] status write failed
  (non-fatal): name 'write_json_atomic' is not defined` (NameError in the
  lane-a status write path, observed 2026-07-02 14:06).
