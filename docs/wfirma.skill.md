# wFirma Product Master & Description Skill

**Status:** active rule. Applies to every flow that creates, references, or
prints a wFirma product, a PZ/customs description, or a wFirma proforma.

**Stopping rule:** If any code path is about to generate a product
description, stop and re-read this file. There is exactly one description
source. New wording is a bug.

---

## 0. Bottom line

wFirma product master is created from the **same locked bilingual
description block** generated during invoice/PZ translation. Proforma and
PZ must reuse that description and **must never** generate a separate
product wording.

---

## 1. Core principle

```
ONE PRODUCT_CODE = ONE LOCKED DESCRIPTION_BLOCK
```

A `product_code` (e.g. `EJL/25-26/1274-3`) carries one and only one
description block. Once written, the block is the source of truth for:

- wFirma product master (`<name>` and `<description>` on `goods/add`)
- PZ / customs goods description PDF
- wFirma proforma line text
- wFirma final invoice line text (when conversion runs)

If a downstream consumer needs different wording, the answer is **fix the
locked block**, not generate a parallel one.

---

## 2. Identity model

| Field | Owner | Used for |
|---|---|---|
| `product_code` | wFirma + PZ app | wFirma `<code>` field; primary join key |
| `design_no` | PZ app only | Local traceability, scan reconciliation, packing audit. wFirma does not own it. |
| `scan_code` | PZ app only | Physical item identity (one row per piece in `inventory_state`) |
| `wfirma_product_id` | wFirma | Returned by wFirma `goods/add` or `goods/find`; persisted locally on `wfirma_products.wfirma_product_id` |

**Never** push `design_no` into the wFirma product payload. **Never**
reverse-derive `product_code` from `design_no` outside the existing
`v_sales_to_wfirma` view.

---

## 3. Description source — single point of truth

The bilingual description block has the existing customs format:

```
Co to za towar:        <description_pl>
Z jakiego materiału:   <material_pl>
Do czego służy:        <purpose_pl>
```

Generated by the translation stage (today: [`polish_description_generator.py`](../polish_description_generator.py)
via `ITEM_TRANSLATIONS[item_type]` lookup at lines 21-91). The four-tuple
`(name_pl, description_pl, material_pl, purpose_pl)` IS the description
block.

Rules:
- Generated **once** at invoice/PZ translation time.
- Stored locally, keyed by `product_code` (or by `item_type` if the
  per-code override is empty — see §3a).
- Reused by every consumer.
- **Never** regenerated by proforma create, product create, or invoice
  conversion code paths.

### 3a. Per-code vs per-type binding

The current translation table at `polish_description_generator.py:21-91`
is keyed by `item_type` (RING, EARRINGS, etc.) — not by `product_code`.
So today, every product whose `item_type=RING` shares the same
description block.

This is acceptable as a default: the customs PDF works, and wFirma's
`<name>` field can reasonably read `"Pierścionek (EJL/25-26/1274-3)"` —
type description plus product_code as differentiator.

When an operator needs unique wording for a specific product_code, an
**override slot** must exist (see §10 future implementation). Until then,
the rule is "per-code identity, per-type description, append product_code
to disambiguate."

## 4. Product creation rule (operator-approved only)

The flow is search → confirm → create:

```
1. Operator triggers create for a product_code.
2. Backend calls wfirma_client.get_product_by_code(product_code).
3. If found:
     - Save mapping locally (PUT /wfirma/products/{code} with
       wfirma_product_id from search).
     - sync_status = matched.
     - Stop. Do NOT call goods/add.
4. If missing:
     - Build payload from the locked description block (see §5).
     - Call wfirma_client.create_product(...) (currently NotImplementedError;
       must be implemented before this branch is reachable in prod).
     - On success: save mapping locally with returned wfirma_product_id.
     - sync_status = matched.
     - Persist the description block on the local row.
5. If create returns error:
     - Do not save a partial mapping.
     - Surface error to operator.
     - Operator decides retry or manual creation in wFirma's own UI.
```

No automatic creation from preview. No bulk create without per-code
operator confirmation. Local mapping never written without a real
`wfirma_product_id` (no fake mappings).

## 5. Product creation payload (locked field set)

The payload **must** be derived from local state only:

```
code        = product_code                            # primary identity (wFirma <code>)
name        = description_line                        # Polish-first / English-after-slash
unit        = "szt."                                  # locked default
vat_rate    = "23"                                    # locked default (jewelry standard)
type        = "good"                                  # locked default; warehouse-tracked
description = description_block, full bilingual three-section text:

  Co to za towar / What is this:    <description_pl> / <description_en>
  Z jakiego materiału / Material:   <material_pl>
  Do czego służy / Purpose:         <purpose_pl>
```

Concrete example (real wFirma output):

```
code:        EJL/25-26/1274-3
name:        Pierścionek ze złota próby 585 z diamentami laboratoryjnymi /
             Lab Grown Diamond Studded 14KT Gold Jewellery RING
description: <full three-section bilingual block>
```

The Polish half is **customs-grade** — same wording as the customs/PZ
description PDF, derived by `customs_description_engine.normalize_item_description(description_en, item_type)` whenever an English invoice description is available. It is NOT the generic `Biżuteria — pierścionek` ITEM_TRANSLATIONS default; that default exists only as the last-resort fallback when no English text is supplied to seed the customs normaliser.

Rationale: the wFirma `<code>` field already holds the product_code —
appending it to `<name>` is noise. The visible product name shown on
proformas, invoices, and operator screens is the locked
Polish-first/English-after-slash composed line straight from
`description_engine`, identical to what the customs PDF prints.

Fallback chain when generating a fresh block:
  `customs_description_engine.normalize_item_description (rich)`
    → `ITEM_TRANSLATIONS[item_type]` (generic)
    → `DEFAULT_TRANSLATION` (last resort)

Fallback chain when `description_line` is empty (operator manual
override with empty fields): `description_line → name_pl → product_code`.

Caller-supplied values (request body) **must not** override any of
these. `code` and `description` come from local state. `name` is
derived deterministically from the locked block.
`unit`/`vat_rate`/`type` are constants.

## 6. Proforma rule

A proforma in this system **is** a wFirma proforma. Identity ladder:

```
PZ app prepares: client + product_code + design_no + qty + unit_price + currency
                       │
                       ▼
       wFirma proforma uses:  product_code + qty + unit_price + currency
       wFirma proforma DROPS: design_no
```

`design_no` traceability stays local in `proforma_drafts.source_lines_json`
([proforma_invoice_link_db.py](../service/app/services/proforma_invoice_link_db.py)).

The proforma create endpoint **must not** generate any product
description. It uses the existing wFirma product as identified by
`product_code`. If the product does not exist in wFirma, the proforma
create is blocked — operator runs the product create flow first
(§4).

## 7. PZ rule — canonical import path

### 7a. Permanent architecture

```
Import invoice + SAD
  → pz_import_processor.py produces:
      product_code, unit_netto_pln (landed cost = FOB + freight + A00 duty)
  → products/resolve maps product_code → wfirma_product_id
  → pz_preview confirms ready=true (all codes mapped, no price conflicts)
  → pz_create sends PZRequest to create_warehouse_pz()
      PZLine.price  = unit_netto_pln   ← import landed cost, NEVER sales price
      PZLine.good_id = wfirma_product_id from wfirma_products table
      contractor_id = WFIRMA_SUPPLIER_CONTRACTOR_ID (import supplier)
  → wFirma stock exists → proforma and final invoice can proceed
```

### 7b. Recovery-only path (NOT the normal flow)

`build_pz_request_from_proforma_snapshot()` in
[`proforma_to_invoice.py:424`](../service/app/services/proforma_to_invoice.py)
and the `--create-pz-before-invoice` flag in
[`send_wfirma_proforma_to_invoice_live_test.py`](../service/app/tools/send_wfirma_proforma_to_invoice_live_test.py)
exist **only** as a recovery workaround for shipments where the
import calculation PZ was not created first.

**Do not use `build_pz_request_from_proforma_snapshot` as a normal path.**
Using proforma `unit_price` (sales price) as the PZ cost basis is
architecturally wrong — it mixes sales pricing into the import cost record.

Normal flow never calls `build_pz_request_from_proforma_snapshot`.

### 7c. Price separation (immutable rule)

| Document | Price field | Source |
|---|---|---|
| wFirma PZ (`PZLine.price`) | `unit_netto_pln` | PZ engine landed cost (FOB + freight + A00 duty) |
| wFirma proforma (`ReservationLine.unit_price`) | `unit_price` | Sales/invoice price in USD, converted by FX |
| wFirma final invoice | Same as proforma | Inherited from proforma snapshot |

These two price fields **must never be swapped or conflated**.

### 7d. Description rule (unchanged)

PZ / customs description PDF uses the same locked description block.
[`polish_description_generator.py:288-317`](../polish_description_generator.py)
must continue reading from the same source as the wFirma product master.

PZ **must not** generate a different description if a product description
already exists in `wfirma_products.description_block` (or wherever the
storage lands per §10).

## 8. Safety gates

Every product create call passes these gates in order. Any miss = abort.

1. **Operator trigger present.** Never fire from preview, never fire from
   proforma create, never fire from a scheduled job.
2. **Search first.** Call `get_product_by_code(product_code)`. If hit,
   skip create entirely.
3. **No duplicate.** UNIQUE(`product_code`) on `wfirma_products` —
   enforced at DB layer.
4. **Persist `wfirma_product_id` on success.** Otherwise the next call
   re-creates and pollutes wFirma master data.
5. **Set `sync_status='matched'` only after the wFirma id is stored.**
   Pending/failed status remains until then.
6. **Keep `design_no` local.** Never include in the wFirma payload.

---

## 9. Forbidden behaviors

- Auto-create product during proforma preview.
- Auto-create product during proforma create.
- Different wording between PZ description, product master, and proforma.
- Master-data creation without operator confirmation.
- Using raw `product_code` (e.g. `EJL/25-26/1274-3`) as the user-visible
  product name on its own — operator-facing wording must come from the
  description block.
- Creating products via bulk endpoint without per-code operator review.
- Saving a local mapping with empty/null `wfirma_product_id`.
- Pulling description text from `design_no`.
- Using `build_pz_request_from_proforma_snapshot` as a normal code path
  (recovery workaround only — see §7b).
- Setting `PZLine.price` from `unit_price` (sales) instead of
  `unit_netto_pln` (import landed cost) — see §7c.
- Calling `create_warehouse_pz` before `pz_preview` returns `ready=true`.

## 11. When this skill applies

Re-read before you:
- Touch `polish_description_generator.py`.
- Touch `wfirma_client.create_product` or `_build_proforma_xml`.
- Add or modify any product description anywhere.
- Add a route under `/api/v1/wfirma/goods/*` or `/api/v1/proforma/*`.
- Change `wfirma_products` or `proforma_drafts` schema.
- Implement `pz_create` route — re-read §7 price separation rules.
- Touch `build_pz_request_from_proforma_snapshot` — read §7b first.

If the change touches description text or product master semantics, the
locked rule wins — even over operator convenience or wFirma payload
shape preferences.
