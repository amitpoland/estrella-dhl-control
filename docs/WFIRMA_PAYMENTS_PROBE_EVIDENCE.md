**Phase 10B planning conclusions that are SAFE to draw from this run:**

- `payments/find` accepts no-filter, `contractor_id` filter, `invoice_id`
  filter, and `date` ge/le filters; status `OK` on every variant.
- `payments/get/{id}` works with the path-id pattern and returns the
  same leaf-tag set as `payments/find`.
- The payment-element schema includes `id`, `value`, `value_pln`,
  `currency_label`, `currency_date`, `currency_exchange`, `date`,
  `payment_method`, `payment_type`, `type`, `contractor/id`,
  `invoice/id`, `expense/id`, `compensation_*`, `interest_*`, `notes`,
  `tags`, plus a number of payroll-only fields (`health`, `social`,
  `labor_fund`, `transitional_retiring_fund`, `staff_salary_content/id`)
  that are out of scope for sales-side Statement of Account.
- Aggregations for a Statement should filter by `invoice/id` non-empty
  to drop payroll / declaration / expense-only payments.

**What this run does NOT settle (re-run required):**

- Whether invoice responses carry `paymentstate`, `alreadypaid`,
  `remaining`, `paymentdate`, `paid_date`. The `invoices/get` probe
  must run against a real invoice id before Phase 10B aggregator code
  can rely on those fields.
