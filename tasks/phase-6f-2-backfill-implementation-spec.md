## 3 — Idempotency keys (final)

- **Charges:** `sha1("legacy_psc:" + batch_id + ":" + client_name + ":" + charge_type)`
  Stored as prefix in `charges.notes`: `[backfill:sha1=<hash>]\n<original note>`
  Detection on re-run: `LIKE '[backfill:sha1=<hash>]%'` (read-only query).
- **Postings:** `BACKFILL-<sha1("legacy_psc_posting:" + batch_id + ":" + client_name)[:16]>`
  Stored in `postings.wfirma_invoice_id`. Detection on re-run: equality match.

## 4 — Eligibility classification

```
classify_row(row) →
  "blocked", "empty_currency"                  if not currency
  "blocked", "non_iso_currency:<x>"            if not ISO 4217
  "blocked", "unknown_charge_type:<x>"         if charge_type ∉ {freight, insurance}
  "blocked", "empty_batch_id"                  if not batch_id
  "blocked", "empty_client_name"               if not client_name
  "blocked", "amount_parse_error:<msg>"        if Decimal parse fails
  "skipped_zero", None                         if amount_minor == 0
  "eligible", None                             otherwise
```

## 5 — Monetary safety

```
amount_minor = int(
    (Decimal(str(amount)) * 100)
    .quantize(Decimal("1"), rounding="ROUND_HALF_EVEN")
)
```

No naive `int(amount * 100)`. Banker's rounding for half-values. Contract test pins `Decimal` usage and forbids `amount * 100` regex.

## 6 — Mixed-currency-in-group policy

If a `(batch_id, client_name)` group has charges with multiple currencies, the live insert raises `ValueError("mixed currencies in group ...")`. This surfaces a data anomaly that the legacy `UNIQUE(batch_id, client_name, charge_type)` constraint allowed but the new schema's synthetic 1-per-tuple posting cannot represent.

In dry-run mode the issue is reflected via `blocked_reasons` (not implemented in 6F.2.a — surface added in 6F.2.b's dry-run report on production data if needed).

## 7 — Chunking

Default 100 rows per chunk. One transaction per `(batch_id, client_name)` group, regardless of group size. Within a group, all charges + the synthetic posting commit atomically. Across groups, each commit is independent.

For typical operator scale (~200-1000 source rows), the entire run is ≪ 10 transactions.

## 8 — Snapshot

`take_snapshot(target_db, snapshot_dir)`:
- Creates `snapshot_dir` if missing.
- Filename: `finance_postings.pre-6F2.<YYYYMMDDTHHMMSSZ>.sqlite`
- Uses `shutil.copy2` (preserves mtime).
- If target_db does not yet exist (fresh production), writes an empty
  marker file at the snapshot path so the snapshot field is non-null in
  the report.

## 9 — Rollback

```sql
DELETE FROM charges WHERE source='legacy_backfill';
DELETE FROM postings WHERE wfirma_invoice_id LIKE 'BACKFILL-%';
```

Belt-and-braces: restore from snapshot file. Snapshot location is recorded in the report's `snapshot` field; rollback procedure is documented in the operator approval package generated by 6F.2.c.
