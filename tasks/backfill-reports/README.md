# Backfill reports

> JSON reports generated by `service/scripts/backfill_finance_postings.py`.
> One file per run. Append-only.

## Filename convention

```
<YYYY-MM-DD>-<mode>-<slug>.json
```

- `mode` = `dryrun` or `live`
- `slug` = short kebab-case identifier (`phase-6f-2b-snapshot`, `phase-6f-2d-live`)

## Schema (per file)

The script writes a single JSON object with the following shape:

```jsonc
{
  "started_at":         "ISO-8601",
  "finished_at":        "ISO-8601",
  "mode":               "dry-run" | "live",
  "source_db":          "<path>",
  "target_db":          "<path>",
  "snapshot":           null | "<snapshot path>",
  "chunk_size":         100,
  "source_rows":        0,
  "eligible_rows":      0,
  "blocked_rows":       0,
  "skipped_zero":       0,
  "duplicate_skipped":  0,
  "charges_to_create":  0,
  "postings_to_create": 0,
  "charges_created":    0,
  "postings_created":   0,
  "blocked_reasons":    {"<reason>": <count>, ...},
  "blocked_examples":   [{"id", "batch_id", "client_name", "charge_type", "reason"}, ...],  // up to 10
  "synthetic_postings": [{"batch_id", "client_name", "synthetic_posting_id",
                          "charge_count", "currency", "amount_total_minor",
                          "already_exists"}, ...]
}
```

## Rules

- **Dry-run reports must never include `charges_created > 0` or `postings_created > 0`.**
  If a report has either, the run was a live execution mis-labelled — investigate.
- **Live reports must always include a `snapshot` path.** If `snapshot` is null in a `mode: live` report, fail loudly.
- **Reports are append-only** — once committed, do not rewrite history. Re-runs produce new files.
- The script writes via atomic `.json.tmp` + replace; never produces a half-written report.

## Workflow

1. **Dry-run** the script against a copy of the production legacy DB.
2. **Read the report** — verify counters, blocked reasons, synthetic postings preview.
3. **Operator approves** live run.
4. **Live run** with `--write --snapshot-dir`.
5. **Verify** post-write counts via `GET /api/v1/finance/postings/{posting_id}/breakdown` for sample synthetic posting ids.
