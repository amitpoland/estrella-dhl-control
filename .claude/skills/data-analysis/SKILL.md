# Estrella PZ — Data Analysis Skill

## Quick Start

Before answering any data question:
1. Identify the entity (batch, shipment, customer, proforma, product).
2. Resolve the primary key (see §Identifier Map).
3. Identify the authority database (see §Authority Map).
4. Use the correct calculation path (see §Calculation Rules).

---

## Business Context

Estrella Jewels imports fine jewellery from India (Mumbai) into Poland (Warsaw) via DHL Express.
Each shipment generates:
- Multiple supplier invoices (PDFs from Indian exporters)
- One ZC429/SAD customs declaration (from DHL Customs WAW agency)
- One PZ document (Przyjęcie Zewnętrzne — goods received note posted in wFirma)
- Sales packing lists (customer allocation of the imported goods)
- Proforma invoices to European customers

The system processes these documents, calculates landed costs, posts to wFirma, and notifies via Zoho Cliq.

## Entity Reference

### 1. Batch / Shipment
- **Primary key:** `batch_id` (UUID string)
- **Authority DB:** `documents.db` → `shipment_documents`
- **Cross-system:** `batch_id` appears in ALL databases and all external systems
- **Natural key:** `awb` (DHL Express AWB number, 10-digit string)
- **Document key:** `doc_no` (PZ number from wFirma, e.g. `PZ/001/2026`)
- **Lifecycle:** created on upload → customs → PZ posted → Cliq notified → WorkDrive filed

### 2. Customer (Bill-To)
- **Primary key:** `contractor_id` (wFirma numeric ID, as TEXT)
- **Authority DB:** `customer_master.sqlite` → `customer_master`
- **Natural key:** `nip` (Polish tax ID, 10 digits) or `vat_eu_number`
- **Cross-system:** `contractor_id` links wFirma, `documents.db`, `proforma_links.db`, carrier accounts
- **Subtypes:** bill-to contractor (always), optional ship-to contractor (Shape B)

### 3. Supplier / Exporter
- **Primary key:** `supplier_contractor_id` (wFirma contractor ID)
- **Authority DB:** `suppliers.sqlite` (local cache; synced from wFirma when `wfirma_sync_suppliers_allowed=True`)
- **Natural key:** `invoice_no` prefix (exporter's invoice number)
- **Canonical exporter:** `ESTRELLA JEWELS LLP.`, Mumbai — hardcoded in `pz_import_processor.py`
- **Cross-system:** `supplier_contractor_id` in `shipment_documents` and carrier `doc_package.py`

### 4. Product / Good
- **Primary key:** `product_code` (format: `{invoice_no}-{position}`, e.g. `INV001-1`)
- **Authority DB:** `wfirma_db.sqlite` (wFirma product mappings); `documents.db` → `product_descriptions` (bilingual descriptions)
- **Cross-system:** `product_code` → `wfirma_product_id` (wFirma internal good ID); `hs_code`/`hsn_code` (customs classification)
- **Constraint:** `wfirma_create_product_allowed=False` means no new products created without operator override

### 5. Customs Declaration (ZC429 / SAD)
- **Primary key:** `mrn` (18-char MRN, unique in `customs_declarations`)
- **Authority DB:** `documents.db` → `customs_declarations`
- **Cross-system:** `batch_id`, `awb`, `related_mrn` in `shipment_documents`
- **Source of truth for:** `duty_pln` (A00 field), `total_cif_usd`, importer identity, exporter identity, invoice reference list

### 6. Invoice (Supplier)
- **Primary key:** `invoice_no` (supplier document number, e.g. `EJ/001/2026`)
- **Authority DB:** `documents.db` → `shipment_documents` (with `document_type='invoice'`) + `invoice_lines`
- **Cross-system:** `invoice_no` appears in ZC429 `invoice_refs`, wFirma doc description, `product_code` generation

### 7. PZ Document
- **Primary key:** `doc_no` (wFirma PZ number, e.g. `PZ/001/2026`)
- **Authority DB:** `documents.db` → `pz_documents`
- **Cross-system:** `batch_id`, `workdrive_pdf_id`, `workdrive_xlsx_id`
- **Source of truth for:** `total_net_pln`, `total_gross_pln`, `duty_a00_pln`, `line_count`, `verification_status`

### 8. Proforma Invoice (Sales)
- **Primary key:** `proforma_id` (wFirma proforma ID)
- **Authority DB:** `proforma_links.db` (shared between `proforma_sync_db.py` and `proforma_draft_sync.py`)
- **Cross-system:** `batch_id`, `contractor_id`, `series_id`
- **States:** `draft → editing → approved → posting → posted → cancelled → superseded`

### 9. Sales Packing List
- **Primary key:** `id` (UUID) in `sales_documents`
- **Authority DB:** `documents.db` → `sales_documents` + `sales_packing_lines`
- **Cross-system:** `batch_id`, `client_name`, `client_ref`, `product_code`, `design_no`, `bag_id`

### 10. AWB Document
- **Primary key:** `awb` (10-digit DHL AWB number)
- **Authority DB:** `documents.db` → `awb_documents`
- **Cross-system:** `batch_id`, `carrier`, `pieces`, `weight_kg`

Full catalog: see `METRIC_CATALOG.md` in this directory.

| Metric | Formula | Authority |
|---|---|---|
| FOB USD | Invoice face value | Supplier invoice |
| Freight USD | `dhl_line_usd × 0.50` | `pz_calculator.py` |
| Insurance USD | `fob_usd × 0.005` | `pz_calculator.py` |
| CIF USD | `fob_usd + insurance_usd + freight_usd` | `pz_calculator.py` |
| Duty PLN | A00 field from ZC429 | ZC429/SAD document |
| Landed USD | `cif_usd + duty_usd` | `pz_calculator.py` |
| Landed PLN | `landed_usd × usd_pln` | `pz_calculator.py` |
| VAT PLN | `landed_pln × 0.23` | `pz_calculator.py` |
| Brutto PLN | `landed_pln + vat_pln` | `pz_calculator.py` |
| Razem Netto PLN | `Σ landed_pln across lines` | `process_batch()` return |
| Razem Brutto PLN | `Σ brutto_pln across lines` | `process_batch()` return |

## Calculation Rules (Never Violate)

1. **`process_batch()` is the ONLY calculation path.** Never recompute in routes, services, Cliq layer, or UI.
2. **Duty always comes from ZC429 A00 field.** Never assume a fixed duty rate.
3. **Freight allocated proportionally by FOB value**, not by piece count.
4. **Insurance = 0.5% of FOB USD** per line.
5. **CIF = FOB + Insurance + Freight** (all in USD before currency conversion).
6. **NBP rate** is fetched live at batch time or supplied manually; never hardcoded in output.
7. **B00 VAT** is reference-only; NOT included in landed cost.
8. **DHL share** = 50% of DHL cost allocated to freight (other 50% to insurance for CIF).

## Analysis Workflow

For any data question:

```
1. What is the entity? (batch / customer / product / proforma / PZ)
2. What is the primary key? (batch_id / contractor_id / product_code / doc_no)
3. What database is authoritative? (see §Database Inventory)
4. Is this a calculated metric? → Use process_batch() output, never recalculate
5. Is this a document field? → Read from documents.db (three-layer priority)
6. Is this a customer field? → Read from customer_master.sqlite
7. Is this a wFirma state? → Check wfirma_db.sqlite cache first; live API only if stale
```

## Common Analysis Questions & Answers

**Q: What is the total landed cost for a shipment?**
A: `pz_documents.total_net_pln` (from `documents.db`); confirmed in `process_batch()` return `total_net`.

**Q: What is the duty paid?**
A: `customs_declarations.duty_pln` (from `documents.db`); same as ZC429 A00 field.

**Q: What exchange rate was used?**
A: `process_batch()` return `nbp.usd_rate`; stored in PZ audit file as `usd_pln`.

**Q: How many pieces were imported?**
A: `pz_documents.line_count` (PZ lines) or `awb_documents.pieces` (carrier declared). These may differ if one line = multiple product types.

**Q: Which customer received what goods?**
A: Join `sales_packing_lines.client_name` + `product_code` + `design_no` in `documents.db`.

**Q: What is the landed cost per piece?**
A: `landed_pln / qty` per `invoice_lines` row; computed inside `process_batch()` as `landed_per_pc_pln`.

**Q: Is the shipment verified against customs?**
A: `pz_documents.verification_status` + `pz_documents.amendment_flags`.

## Known Pitfalls

1. **Do not use `pz_calculator.py` constants for duty.** `DUTY_RATE = 0.12` is a golden-test fallback only; real duty comes from ZC429 A00.
2. **`customer_master.upsert_customer` is FULL-SET semantics.** A partial update silently wipes all unset fields. Use `upsert_identity_only` for wFirma-sourced fills.
3. **Finance postings use integer cents (minor units), NOT float.** Never sum floats from `finance_postings.sqlite`.
4. **`product_descriptions` is the authoritative bilingual source.** Do not derive product names from invoice descriptions.
5. **`customers_invoice_snapshot.db` is a cache.** It reflects wFirma state at last sync, not real-time. Monetary values are stored as Decimal-as-string TEXT.
6. **Two ship-to shapes for customers:** Shape A = alternate address on same wFirma contractor; Shape B = separate wFirma contractor with its own `contractor_id`.
7. **`batch_id` is not the same as `awb`.** A batch can have one or more AWBs. The canonical link is `shipment_documents.awb`.
8. **HSN codes (7113...) are NEVER valid as quantities.** `pz_import_processor.py` explicitly blocks known HSN codes from being parsed as piece counts.
9. **`carrier_shipments` in live mode rejects submissions.** Only shadow mode submissions are accepted automatically; live requires `carrier_live_allowlist`.
10. **Proforma service charges (`freight`, `insurance`) are operator-entered**, not engine-calculated. They live in `proforma_service_charges_db.py`.

## Verification Checklist

Before reporting any metric or data point:
- [ ] Have you identified the authoritative database (not a cache or derived source)?
- [ ] For calculated metrics: did you use the `process_batch()` return, not a manual recalculation?
- [ ] For duty: did you read ZC429 A00, not apply `DUTY_RATE = 0.12`?
- [ ] For customer identity: did you read `customer_master.sqlite`, not wFirma API directly?
- [ ] For monetary values in `finance_postings.sqlite`: are you working in integer cents?
- [ ] For product names: did you read `product_descriptions`, not `invoice_lines.description`?
- [ ] Have you checked `pz_documents.verification_status` for data quality signals?
- [ ] Have you checked `pz_documents.amendment_flags` for known discrepancies?
